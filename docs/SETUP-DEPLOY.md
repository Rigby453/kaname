# Как выложить «Главное» в интернет (деплой) — пошагово

> Гайд для не-разработчика. Делай по шагам сверху вниз, ничего не пропуская.
> В конце у тебя будет:
> - **Лендинг** (сайт-витрина): `https://rigby453.github.io/glavnoe/`
> - **Веб-приложение** (рабочее приложение в браузере): `https://rigby453.github.io/glavnoe/app/`
> - **Бэкенд** (сервер, который хранит данные и логинит пользователей) — на Render.

Из чего состоит запуск (3 части):

1. **Бэкенд на Render** — сервер. Без него вход и сохранение данных не работают. (Раздел A)
2. **GitHub Pages** — бесплатный хостинг для лендинга и веб-приложения. (Разделы B, C, D)
3. **Связка** — приложение в браузере должно знать адрес сервера, а сервер — разрешить запросы с сайта. (Разделы A и B вместе)

Порядок важен: **сначала бэкенд (A)**, потому что его адрес понадобится в разделе B.

---

## Раздел A — Бэкенд на Render

> Подробные значения секретных переменных тебе даст отдельная инструкция/агент по бэкенду.
> Здесь — только клики по порядку.

1. Зайди на **https://render.com** и создай аккаунт (можно войти через GitHub — удобнее всего).
2. Нажми **New** (вверху справа) → выбери **Blueprint**.
3. **Подключи репозиторий** `Rigby453/glavnoe` (если просит — разреши Render доступ к GitHub).
4. Render сам найдёт файл **`render.yaml`** в репозитории и предложит создать по нему сервисы. Подтверди.
5. Render попросит ввести **секретные переменные окружения** (Environment Variables) в дашборде.
   Введи их по инструкции бэкенда. Среди них обязательно будет переменная:

   ```
   ALLOWED_ORIGINS = https://rigby453.github.io
   ```

   > ВАЖНО. Это адрес, с которого браузерному приложению разрешено обращаться к серверу.
   > Если тут не будет `https://rigby453.github.io`, то веб-приложение **не сможет залогиниться**
   > (браузер заблокирует запросы — это называется CORS-ошибка). Без `/glavnoe` на конце, только домен.

6. Нажми **Deploy** и дождись, пока статус станет зелёным (Live). Первый запуск может занять несколько минут.
7. **Скопируй публичный адрес** сервиса — он выглядит так:

   ```
   https://имя-сервиса.onrender.com
   ```

   Этот адрес понадобится в **Разделе B**. Запиши его.

---

## Раздел B — Сказать сайту адрес сервера (GitHub Variable)

Веб-приложение собирается заранее, поэтому адрес сервера нужно «зашить» в него при сборке.
Это делается через одну переменную в настройках репозитория.

1. Открой репозиторий на GitHub: **https://github.com/Rigby453/glavnoe**
2. Вкладка **Settings** (вверху).
3. Слева: **Secrets and variables** → **Actions**.
4. Сверху переключись на вкладку **Variables** (не Secrets!).
5. Нажми **New repository variable**.
6. Заполни:
   - **Name:** `API_BASE_URL`
   - **Value:** адрес Render из Раздела A, например `https://имя-сервиса.onrender.com`
     (без слэша на конце).
7. Нажми **Add variable**.

> Если эту переменную не задать, приложение соберётся с адресом `http://localhost:3000`
> (то есть будет искать сервер на твоём компьютере) и в интернете логиниться не сможет.
> После того как зададите/измените переменную — нужно пересобрать сайт (Раздел D).

---

## Раздел C — Включить GitHub Pages

1. В том же репозитории: **Settings** → слева **Pages**.
2. В поле **Source** выбери **GitHub Actions** (НЕ «Deploy from a branch»).
3. Больше ничего настраивать не нужно — сохранять отдельной кнопкой обычно не требуется.

---

## Раздел D — Запустить сборку и публикацию

Сборка запускается автоматически при каждом изменении в ветке `main`. Но первый раз
(или после смены переменной из Раздела B) её удобно запустить руками:

1. Репозиторий → вкладка **Actions** (вверху).
2. Слева выбери workflow **«Deploy web (landing + Flutter app) to GitHub Pages»**.
3. Справа кнопка **Run workflow** → выбери ветку `main` → **Run workflow**.
4. Подожди, пока появятся зелёные галочки у задач **build** и **deploy** (несколько минут —
   сборка Flutter небыстрая).

Когда всё зелёное, сайт уже опубликован:
- Лендинг: **https://rigby453.github.io/glavnoe/**
- Веб-приложение: **https://rigby453.github.io/glavnoe/app/**

> Совет: если поменял переменную `API_BASE_URL` (Раздел B), обязательно перезапусти workflow,
> иначе приложение останется со старым адресом сервера.

---

## Раздел E — Проверка, что всё работает

1. Открой **https://rigby453.github.io/glavnoe/** — должен открыться лендинг (витрина).
2. Нажми кнопку **«Open the Web App»** (или ссылку «Web App» в подвале) —
   должно открыться **https://rigby453.github.io/glavnoe/app/** с приложением.
3. Попробуй **зарегистрироваться / залогиниться**.
   - Если вход проходит и данные сохраняются — всё связано правильно. 🎉
   - Если вход не идёт, открой в браузере консоль (клавиша F12 → вкладка Console) и посмотри
     ошибки:
     - Слова **CORS** / «blocked by CORS policy» → проверь `ALLOWED_ORIGINS` на Render (Раздел A, шаг 5):
       там должно быть ровно `https://rigby453.github.io`.
     - Ошибки сети / адрес `localhost` → значит не задана/не применилась переменная `API_BASE_URL`
       (Раздел B) — задай её и перезапусти workflow (Раздел D).

## Раздел F — Про кнопки App Store и Google Play (важно)

На лендинге кнопки **App Store** и **Google Play** пока ведут «в никуда» (это плейсхолдеры).
Они заработают **только после публикации приложения в магазинах**, а для этого нужно:

- **Аккаунт разработчика Apple** (платный, ~$99/год) и **Mac** для сборки iOS-версии.
- **Аккаунт разработчика Google Play** (разовый платёж ~$25).

Это отдельная история, **не связанная с этим деплоем**. Веб-версия (Разделы A–E) работает
полностью без магазинов и аккаунтов разработчика — её достаточно, чтобы дать людям доступ
к приложению по ссылке.
